## 1. Что уже удалось

### 1.1. Рабочий XML (`order_example.xml`)

- XML загружается в обработку **«Ввод нового заказа»** → документ `асСпецификацияЗаказа`.
- Корректно подбираются: продукт по артикулу, материал по коду типа, соединения, цены.
- Формат характеристики строго:

```text
ПАРАМЕТР<значение>ПАРАМЕТР_
```

Примеры:

```text
D0160D0_L0100L0_          → D0=160, L0=100
A0300A0_B0200B0_L0100L0_  → A0=300, B0=200, L0=100
D0160D0_P004P0_           → D0=160, P0=4
```

Парсер — функция `Определение_Значения` из модуля загрузки.

### 1.2. Материал

- В `<material>` передаётся **цифровой код типа материала**, а не код справочника.
- `1` — оцинкованная сталь.
- `2` — нержавеющая сталь.
- `3` — чёрная сталь.
- Толщина берётся из `<thickness>`.

### 1.3. Соединения

- В XML передаётся только **первый символ** значения (функция `Лев(значение,1)`).
- Круглые (`c0..c3`): `1` бандаж, `2` без соединения, `3` ниппель, `4` фланец (кр).
- Прямоугольные (`r0..r3`): `5` фланец (пр), `6` шина / уголок, `7` рейка, `8` без соединения.
- `0` — сторона не используется.

### 1.4. 120 продуктов с паттернами обозначений

Из модуля `модуль ввод нового заказа.txt` выгружены в `product_designation_patterns.json`:

- артикул (`1-1-1`, `12-1-3`, …);
- наименование;
- группа продукции;
- выражение обозначения (1С-строка вида `"Ф"+D_+"-"+L_`);
- список XML-параметров (`D0`, `L0`, `A0`, `B0`, `U0`, `P0` и др.).

Примеры:

| Артикул | Обозначение | Параметры |
|---------|-------------|-----------|
| `1-1-1` | `Ф{D}-{L}` | `D0`, `L0` |
| `1-1-2` | `Ф{D}-{L}` | `D0`, `L0` |
| `1-2-1` | `{A}x{B}-{L}` | `A0`, `B0`, `L0` |
| `12-1-3`| `Ф{D}` | `D0`, `P0` |

### 1.5. Справочники и константы

Сохранены в `catalog_data.json`:

- `асПродукция` — 215 позиций с артикулами, группами, видом сечения.
- `асМатериалы` — ~200 материалов (листы, штрипсы, оцинковка, нерж.).
- `асСоединениеВоздуховодов` — 13 вариантов с видом сечения.
- Константы по умолчанию:
  - круглое соединение — **Без соединения**;
  - прямоугольное соединение — **Шина / уголок**;
  - материал по умолчанию — рулон оцинкованной стали.

## 2. Что показал реальный заказ 000000841

Документ `асСпецификацияЗаказа` № **000000841** создан из `order_example.xml`.

### 2.1. Строки заказа

| № | Артикул | Наименование | D0 | A0 | B0 | L0 | P0 | Материал (факт) | Соединения |
|---|---------|--------------|----|----|----|----|----|-----------------|------------|
| 1 | `1-1-1` | Воздуховод круглого сечения прямошовный | 160 | — | — | 100 | — | Лист.нерж. (AISI 430) 0.70 | c0=Ниппель, c1=Ниппель |
| 2 | `1-1-2` | Воздуховод круглого сечения спирально-навивной | 200 | — | — | 100 | — | Штрипс оц.(08ПС) 0.50 | c0=Ниппель, c1=Ниппель |
| 3 | `1-2-1` | Воздуховод прямоугольного сечения | — | 300 | 200 | 100 | — | Лист.нерж. (AISI 430) 0.70 | r0=Шина/уголок, r1=Шина/уголок |
| 4 | `12-1-3`| Фланец (кр) | 160 | — | — | — | 4 | Лист.нерж. (AISI 430) 0.70 | — |

### 2.2. Проблема «Авторасчёт» подтверждена в базе

В сохранённом документе 000000841 **поле `Обозначение` у всех строк = `Ф160`**:

- Строка 1: должно быть `Ф160-100`, в базе `Ф160`.
- Строка 2: должно быть `Ф200-100`, в базе `Ф160`.
- Строка 3: должно быть `300x200-100`, в базе `Ф160`.
- Строка 4: фланец `Ф160` — совпало случайно.

Это прямое доказательство: если при загрузке в форме стояла галочка **«Авторасчёт»**, обозначения всех строк считаются по параметрам последней строки (заголовочные `D_`, `L_` и т.д. остаются от последней строки). После снятия галочки вручную обозначения становятся правильными.

### 2.3. Материал и соединения могут отличаться от XML

- В XML для всех строк было `<material>1</material>` (оцинкованная сталь) и `<thickness>0.7</thickness>`.
- В сохранённом заказе строки 1, 3, 4 имеют **нерж. AISI 430 0.70**.
- Строка 2 имеет **штрипс оц.(08ПС) 0.50**.
- Соединения в XML были `2` (без соединения) / `6` (шина/уголок), а в базе круглые строки получили **Ниппель**.

Вывод: форма пересчитывает материал и соединения по своим правилам/константам; либо оператор вручную изменил значения после загрузки. Для полной автоматизации нужно либо передавать точные коды материалов/соединений, либо корректно настраивать константы по умолчанию.

## 3. Нерешённые проблемы

### 3.1. Имя реквизита «Авторасчёт»

- Видна галочка на форме обработки, но имя элемента/реквизита не найдено ни в `модуль загрузки с сайта.txt`, ни в `модуль ввод нового заказа.txt`.
- Возможные пути:
  1. Распаковать `MCP_Toolkit.epf` (`v8unpack`) и посмотреть `асВводНовогоЗаказа.Форма.Форма`.
  2. В 1С открыть форму в конфигураторе и посмотреть имя элемента управления.
  3. Если реквизит недоступен через XML, сделать обработку-обёртку, которая открывает форму, снимает галочку программно и вызывает стандартную загрузку.
- Без имени реквизита передать флаг через XML нельзя.

### 3.2. Ошибка `order_variants.xml`

При загрузке вариантного XML возникает:

```text
Преобразование значения к типу Число не может быть выполнено
```

по полю `d2`. Вероятная причина — неверный формат характеристики или пустое/некорректное значение для дополнительных параметров (`d1`, `d2`, `d3` и т.д.).

### 3.3. Сопоставление обязательных параметров для каждого продукта

Попытка прочитать `Справочник.асГруппыПродукции.Параметры` как табличную часть не сработала — такой ТЧ нет. Параметры, скорее всего, хранятся:

- в регистре сведений;
- в табличной части с другим именем;
- или вычисляются динамически по группе и виду сечения.

Паттерны из `product_designation_patterns.json` уже дают список XML-параметров для каждого артикула, но нужно убедиться, что это полный набор.

| `P0`, `p1..p3` | Параметры P |

## 5. Что уже сделано на базе исследования

- Создана полная таблица сопоставления `product_article_mapping.json` для 120 продуктов.
- Написан генератор XML `generate_order_xml.py`, который по структурированным данным (список строк заказа) формирует файл, совместимый с загрузчиком 1С.
- Реализован сопоставитель текстового описания → артикул 1С (`match_description_to_product.py`).
  - Использует нечёткое сравнение наименований + ключевые слова (круглый/прямоугольный, прямошовный/спиральный, фланец/отвод/переход и т.д.).
  - Извлекает размеры: диаметр `D0`, стороны `A0`/`B0`, длину `L0`, количество отверстий `P0`.
  - Проверено на описаниях: "Воздуховод круглый прямошовный D160 L1000", "Воздуховод прямоугольный 300x200 L1000", "Фланец круглый Ф160 на 4 отверстия", "Отвод круглый 90 градусов D160", "Переход круглый с D200 на D160", "Шумоглушитель прямоугольный 500x300 L1000" — все определены корректно.
- Создан полный конвейер `process_specification_table.py` для загрузки таблиц спецификаций (CSV/Excel).
  - Определяет круглое/прямоугольное сечение по размеру (`100` → круглое, `300x200` → прямоугольное).
  - Переводит метраж в штуки по стандартным длинам (круглые 3000 мм, прямоугольные 1250 мм — настраивается).
  - Распознаёт фасонные изделия: отводы (2-1-1, 2-2-1), переходы (3-1-1, 3-2-1), тройники (4-1-1, 4-2-1), заглушки (6-1-1, 6-2-1), фланцы (12-1-3, 12-2-2), шумоглушители (15-1-1, 15-2-1) и др.
  - Распознаёт КСД-адаптеры из строк диффузоров (`4АПН 600x600 + КСД 200` → артикул 10-2-3).
  - Пропускает агрегатные строки фасонных изделий (без детализации) и диффузоры без артикула, формируя подробный отчёт `*_skipped.json`.
  - Проверен на примере спецификации: 15 строк распознано, 4 позиции пропущены с объяснением.
