### 3. Создание и корректировка шаблонов XML
- Создан базовый шаблон `order_template.xml`.
- Создан пример заполненного заказа `order_example.xml`.
- Примеры скорректированы под реальные артикулы из справочника `асПродукция` (`1-1-1`, `1-1-2`, `1-2-1`, `12-1-3`).
- **Установлен правильный формат характеристики:** `D0160D0_L01250L0_`, `A0300A0_B0200B0_L01000L0_` (парсер `Определение_Значения`).
- **Исправлено заполнение материала:** в `<material>` передаётся полный артикул материала с толщиной (`01-02-01-01-0.70`), как в выгрузке реального заказа 000000839, а не код справочника.
- **Настроены соединения:** для активных сторон используются корректные ID (`2` — без соединения для круглых, `6` — шина/уголок для прямоугольных). Для неиспользуемых/неприменимых сторон (`c2/c3`, `r2/r3`, фланец `12-1-3`) допустимо значение `0`, равнозначное пустой ссылке, как в реальном заказе 000000839.

### 4. Диагностика ошибок загрузки
- Выяснена причина ошибки «Преобразование значения к типу Число не может быть выполнено»: пустой тег `<thickness>`.
  - Решение: всегда заполнять `<thickness>` числом, даже `0` для неметаллических позиций.
- Выяснена причина ошибки «Номенклатура или вид номенклатуры не заполнен или заполнен неверно`: в XML использовались несуществующие артикулы.
  - Установлено, что поиск продукта идёт по `асПродукция.Продукт.Артикул`, а не по `асПродукция.Артикул`.
- Проанализирована функция `Определение_Значения`: характеристика парсится по параметрам `D0..D4`, `A0..A5`, `B0..B5`, `L0..L6` и др. Формат: `ПАРАМЕТР<значение>ПАРАМЕТР_`.
- Проанализирована функция `ЗаказыtПриИзмененииНаСервере`: после поиска материала по типу выполняется поиск материала того же родителя с толщиной из `<thickness>`.

## Текущие проблемы и ограничения

1. **MCP-сервер не строит поисковые индексы**: в каталоге платформы `D:\Program Files\1cv8\8.3.27.1989` отсутствует файл `shcntx_ru.hbk`. Для полноценного поиска по справке нужна полная установка 1С с толстым клиентом.
2. **Логика общего модуля `асОбщийМодуль` недоступна для чтения**: функции `НахождениеМатериалаПоТипу` и `НахождениеСоединениеПродуктаПо_Id_c` / `_Id_r` находятся в общем модуле, который нельзя открыть без конфигуратора. Значения материала и соединений подобраны эмпирически на основе справочников.
3. **Обозначения и цены требуют проверки на реальной загрузке**: после исправления характеристик расчёт цен запускается, но обозначения для строк, загруженных с сайта, формируются через заголовочные реквизиты и могут потребовать доработки формы обработки.

## Результаты тестовой загрузки (импорт order_example.xml)

- **Материал / толщина:**
  - Полный артикул (`01-02-01-01-0.70`) не подошёл — толщина сбросилась на `0.50`.
  - Префикс артикула (`01-02-01-01`) тоже не подошёл — та же ошибка, поле «Материал» осталось пустым.
  - **Вывод:** функция `НахождениеМатериалаПоТипу` ожидает цифровой код типа материала (`1` — оцинк., `2` — нерж., `3` — черн.), как в тестовой выгрузке модуля «Ввод нового заказа».
- **Соединения:** определились корректно (`/` для «Без соединения», `ш20у95/ш20у95` для «Шина/уголок»).
- **Обозначения:** все строки получили одинаковое обозначение последней строки (`Ф160`). Причина — в процедуре `РедактироватьВсеНаКлиенте_ДляИзмененных()` не устанавливаются заголовочные реквизиты из текущей строки перед расчётом обозначения. Требуется доработка кода формы обработки.
- **Параметр `p` для фланца:** добавлен в XML (`D0160D0_P004P0_`), требуется проверка после устранения проблемы с обозначениями.

- **Соединения:** определились корректно (`/` для «Без соединения», `ш20у95/ш20у95` для «Шина/уголок»).
