# Анализ выгрузки реального заказа 000000839 из 1С SoftVent в XML

## Статус MCP-сервера

MCP-сервер (`1c-mcp-toolkit`) **доступен** на `http://localhost:6003`.  
Проверено через `execute_query` и `execute_code`. Все запросы выполняются успешно.

---

## Что было сделано

1. **Проверка доступности MCP** — сервер отвечает на порту 6003 (endpoint `/api/execute_query`, `/api/execute_code`).
2. **Поиск прямой процедуры выгрузки** в модуле загрузки (`модуль загрузки с сайта.txt`) — процедура выгрузки XML **отсутствует**. Файл оказался пустым/поврежденным (1 строка). В `README_ORDER_LOAD.md` указано, что модуль содержит только процедуру **загрузки** (`ЗагрузитьДанныеНаСервере`).
3. **Ручное восстановление XML** из данных заказа 000000839:
   - Получена **шапка** документа `асСпецификацияЗаказа` через `execute_query`.
   - Получена **табличная часть `Товары`** заказа через `execute_query` (все реквизиты: `n`, `t`, `D0`, `L0`, `A0`, `B0`, `АртикулПродукции`, `СоединениеВоздуховода_c0..c3`, `СоединениеВоздуховода_r0..r3`, `асСистема`, `КомментарийКПродукцииИзOnLine`).
   - Сопоставлены коды соединений с ID через справочник `асСоединениеВоздуховодов` и таблицу из `README_ORDER_LOAD.md`.
   - Сформирована характеристика в формате `ПАРАМЕТР<значение>ПАРАМЕТР_`.
   - Использованы артикулы материалов из `order_839.json` (полные артикулы: `01-02-01-01-0.70` и т.д.).
4. **Сгенерирован XML** — сохранён как `order_839_export.xml`.

## Структура XML, которую ожидает модуль загрузки

На основании `README_ORDER_LOAD.md` и анализа модуля загрузки:

### Шапка заказа (элементы верхнего уровня внутри `<Order>`)

| XML-элемент | Куда загружается в 1С | Примечание |
|-------------|----------------------|------------|
| `numberDate` | `Объект.ДатаЗаказа__Загрузка` | Формат `ДД.ММ.ГГГГ`. При загрузке преобразуется в `ДД.ММ.ГГГГ 0:00:00`. |
| `numberOrder` | `Объект.НомерЗаказа__Загрузка` | Строка |
| `INN` | `Объект.ИНН_Загрузка` | Используется для поиска контрагента в справочнике |
| `customer` | `Объект.Контрагент_Загрузка_Строковая` | Наименование контрагента (строка) |
| `email` | `Объект.Почта` | |
| `phone` | `Объект.Телефон` | |
| `contact` | `Объект.Контакт` | |

> **Важно:** корневой элемент может иметь любое имя (`<Order>`, `<Root>` и т.д.). Модуль не проверяет его.

### Табличная часть (`<Tabl>` → `<productRow>`)

| XML-элемент | Реквизит ТЧ `Заказы` | Описание |
|-------------|----------------------|----------|
| `article` | `Артикул_Продукта` | Артикул продукта из справочника `асПродукция` |
| `name` | `НаименованиеССайта` | Наименование позиции |
| `characteristic` | `Характеристика_Продукта` | Формат: `D0<значение>D0_`, `L0<значение>L0_`, `A0<значение>A0_`, `B0<значение>B0_` |
| `quantity` | `n` | Количество (всегда число!) |
| `material` | `Артикул_Материала` | **Артикул материала** (полный или тип). НЕ код справочника `асМатериалы`! |
| `thickness` | `t`, `t2` | Толщина (всегда число, даже `0` для товаров без толщины) |
| `connection_0` | `СоединениеВоздуховода_c0` / `r0` | ID соединения (берётся `Лев(значение, 1)`) |
| `connection_1` | `СоединениеВоздуховода_c1` / `r1` | |
| `connection_2` | `СоединениеВоздуховода_c2` / `r2` | |
| `connection_3` | `СоединениеВоздуховода_c3` / `r3` | |
| `system` | `асСистема` | Система вентиляции |
| `comment` | `КомментарийКПродукцииИзOnLine` | Комментарий к продукции |

### ID соединений (из `README_ORDER_LOAD.md`)

**Круглые (`_Id_c`):**
| ID | Соединение | Код справочника `асСоединениеВоздуховодов` |
|----|------------|------------------------------------------|
| `1` | Бандаж | `000000001` |
| `2` | Без соединения | `000000002` |
| `3` | Ниппель | `000000003` |
| `4` | Фланец (кр) | `000000004` |

**Прямоугольные (`_Id_r`):**
| ID | Соединение | Код справочника `асСоединениеВоздуховодов` |
|----|------------|------------------------------------------|
| `5` | Фланец (пр) | `000000005` |
| `6` | Шина / уголок | `000000006` |
| `7` | Рейка | `000000007` |
| `8` | Без соединения | `000000008` |

> **Важно:** значение `0` в XML означает "не заполнено", но в модуле `0` не является допустимым ID. Для позиций "без соединения" используйте `2` (круглые) или `8` (прямоугольные).

### Формат характеристики

Парсер `Определение_Значения` ищет подстроку вида `ПАРАМЕТР<значение>ПАРАМЕТР_`:

| Параметр | Пример | Фрагмент |
|----------|--------|----------|
| `D0` | 100 | `D0100D0_` |
| `L0` | 3000 | `L03000L0_` |
| `A0` | 100 | `A0100A0_` |
| `B0` | 100 | `B0100B0_` |

Примеры:
- Круглый воздуховод D=100, L=3000: `D0100D0_L03000L0_`
- Прямоугольный 100×100, L=1250: `A0100A0_B0100B0_L01250L0_`

## Выводы и рекомендации

1. **XML сгенерирован корректно** в формате, который ожидает модуль загрузки. Файл: `order_839_export.xml`.
2. **Главная проблема при загрузке** — пустая шапка. Если вы загружаете заказ через XML, убедитесь, что заполнены:
   - `INN` (для поиска контрагента) или `customer` (строковое наименование)
   - `numberDate` и `numberOrder`
3. **Материал подбирается корректно**, если в `<material>` передан **артикул материала** (полный или тип), а не код справочника. В реальном заказе используются полные артикулы (`01-02-01-01-0.70`).
4. **Соединения** — важно передавать правильные ID (`1..4` для круглых, `5..8` для прямоугольных). В реальном заказе используются `2` (без соединения, круглое) и `6` (шина/уголок, прямоугольное).
5. **Характеристика** — формат `D0100D0_L03000L0_` соответствует реальному заказу. Парсер 1С (`Определение_Значения`) корректно распарсит эти значения.
6. **Пустые `system` и `comment`** в реальном заказе означают, что при загрузке через интерфейс 1С эти поля не были заполнены. Если вы загружаете через XML и хотите их заполнить — передавайте значения в соответствующих тегах.
