## 3. Как работает загрузка XML (по данным из README и стека ошибки)

### 3.1. Цепочка вызовов при загрузке

```
[Форма] ВыполнитьЗагрузкуНаСервере(Адрес, ИмяФайла)          → строка 47544
  └─ [Форма] ОпределениеПараметровИзФайлаЗагрузки()          → строка 47583
       └─ [Форма] Цикл по ТЧ "Заказы"
            └─ [асОбщийМодуль] КорректировкаТолщиныМатериалаССайта(Стр.t) → строка 32350
                 └─ [асОбщийМодуль] Модуль, строка 7863
                      Запрос.УстановитьПараметр("ТолщинаССайта", Число(ТолщинаССайта));
```

### 3.2. Что происходит в `ОпределениеПараметровИзФайлаЗагрузки()`

Это ключевая процедура, которая:
1. Проходит по каждой строке табличной части `Заказы`.
2. Вызывает `асОбщийМодуль.НахождениеМатериалаПоТипу(Стр.Артикул_Материала, Стр.Продукт)` — ищет материал по артикулу с учётом продукта.
3. Вызывает `асОбщийМодуль.КорректировкаТолщиныМатериалаССайта(Стр.t)` — **корректирует толщину**.
4. Пытается подобрать материал с этой скорректированной толщиной (вызов `ЗаказыtПриИзмененииНаСервере` или аналогичная логика внутри цикла).
5. Если материал не найден — выводит сообщение и устанавливает цену = 0.

## 4. Почему толщина заменяется на 0.50

### 4.1. Анализ сообщений системы

Пользователь видит:
- «По строке:3 толщина металла у Заказчика: 0.7. Заменено на 0.5 или на Корректировочный»
- «Материала с выбранной толщиной: 0.50 нет.»
- «Цена = 0 по строке1»

Это означает, что внутри `ОпределениеПараметровИзФайлаЗагрузки()` есть логика примерно такого вида:

```bsl
// Псевдокод на основе сообщений
ТолщинаССайта = Стр.t; // 0.7
ТолщинаНовая = асОбщийМодуль.КорректировкаТолщиныМатериалаССайта(ТолщинаССайта);

// Если корректировка не вернула исходную толщину — значит, 0.7 не прошла валидацию
Если ТолщинаНовая <> ТолщинаССайта Тогда
    Сообщить("По строке:" + НомерСтроки + " толщина металла у Заказчика: " + ТолщинаССайта + ". Заменено на " + ТолщинаНовая + " или на Корректировочный");
КонецЕсли;

// Пытаемся найти материал с новой толщиной
Материал = НайтиМатериалПоРодителюИТолщине(РодительМатериала, ТолщинаНовая);
Если Материал = Неопределено Тогда
    Сообщить("Материала с выбранной толщиной: " + Формат(ТолщинаНовая, "ЧДЦ=2") + " нет.");
    Цена = 0;
КонецЕсли;
```

**Вывод:** `КорректировкаТолщиныМатериалаССайта(0.7)` возвращает `0.5` вместо `0.7`.

### 4.2. Почему `КорректировкаТолщиныМатериалаССайта` возвращает 0.5?

Скорее всего, функция работает по одному из следующих сценариев:

**Сценарий А — поиск по родителю материала:**
- Функция получает толщину и (внутри себя) ссылку на материал.
- Если материал ещё **не определён** (или определён некорректно), функция не может найти «родственные» материалы с толщиной 0.7.
- Возвращает «минимальную» или «корректировочную» толщину 0.5.

**Сценарий Б — валидация по списку доступных толщин для продукта:**
- Функция проверяет, допустима ли толщина 0.7 для данного продукта (1-1-1, 1-1-2 и т.д.).
- Если продукт не ожидает толщину 0.7 — возвращает 0.5 (минимальную из допустимых).

**Сценарий В — разница в формате числа:**
- В справочнике `асМатериалы` толщина хранится в виде строки в артикуле: `01-02-01-01-0.70`.
- Функция может сравнивать `Стр.t` (число 0.7) со значением из справочника, и при определённом способе сравнения (например, через строковое представление) 0.7 ≠ 0.70.

### 4.3. Почему это происходит для ВСЕХ строк?

В вашем `order_example.xml` **все 4 строки** используют один и тот же материал:
```xml
<material>01-02-01-01-0.70</material>
```

Если проблема в формате `<material>` (например, передача **полного артикула с толщиной** вместо **типа/префикса**), то она проявится на всех строках одинаково.

## 5. Почему материал не подбирается

### 5.1. Ожидаемое значение в `<material>`

По данным `README_ORDER_LOAD.md` и шаблона `order_template.xml`:

| Вариант | Пример | Результат |
|---------|--------|-----------|
| **Тип материала (префикс)** | `01-02-01-01` | Система найдёт группу материалов «Рулон оц.» и подберёт конкретную толщину из `<thickness>` |
| **Полный артикул** | `01-02-01-01-0.70` | Система найдёт конкретный материал с толщиной 0.70 мм |
| **Код справочника** | `000000037` | ❌ НЕ ПОДХОДИТ — `НахождениеМатериалаПоТипу` работает с артикулом, а не с кодом |

Ваш XML использует полный артикул `01-02-01-01-0.70`. Теоретически это должно работать, но на практике **в `order_template.xml` используется именно тип-префикс** (без толщины).

Это неслучайно: когда в `<material>` передаётся **тип-префикс**, функция `НахождениеМатериалаПоТипу` сначала находит «группу» (родителя), а затем корректировщик толщины ищет внутри этой группы нужную толщину. Когда передаётся **полный артикул**, логика может работать иначе — материал находится сразу, но последующий пересчёт толщины может «потерять» родителя или неправильно определить доступные толщины.

### 5.2. Расхождение с рабочим заказом 839

В `order_839.json` материал указан как `01-02-01-01-0.70`, и он работает. **Но есть важное отличие:**

| | Рабочий заказ 839 | Ваш order_example.xml |
|---|-------------------|----------------------|
| Продукт 1 | `1-1-2` (спирально-навивной) | `1-1-1` (прямошовный) |
| Продукт 2 | `1-2-1` (прямоугольный) | `1-1-2` (спирально-навивной) |
| Продукт 3 | `2-2-2` (тройник прямоугольный) | `1-2-1` (прямоугольный) |
| Продукт 4 | `3-1-1` (переход круглый) | `12-1-3` (фланец) |

**В рабочем заказе отсутствует продукт `1-1-1`!**

Возможно, для `1-1-1` (прямошовный круглый воздуховод) в базе настроена **другая группа материалов** (например, `01-01-01-01` — лист оцинкованный, а не рулон `01-02-01-01`). Если вы передаёте рулон для прямошовного воздуховода, система может не найти допустимую толщину и сбрасывать на 0.5.

### 5.3. Соединения: недопустимое значение `0`

В вашем XML:
```xml
<connection_2>0</connection_2>
<connection_3>0</connection_3>
```

По данным анализа, `0` **не является допустимым ID** соединения. Система берёт только **первый символ** значения (`Лев(значение, 1)`). Если первый символ — `0`, то `НахождениеСоединениеПродуктаПо_Id_c("0")` или `НахождениеСоединениеПродуктаПо_Id_r("0")` вернёт `Неопределено`, что может вызывать ошибки в последующей обработке.

**Правильные значения для «без соединения»:**
- Круглые продукты: `2`
- Прямоугольные продукты: `8`

### 5.4. Справочник материалов: разные родители для одного типа

В `catalog_data.json` обнаружено, что материалы группы «Рулон оц.» разбросаны по разным родителям:

| Код | Толщина | Артикул | Родитель |
|-----|---------|---------|----------|
| 000000163 | 0.50 | `01-02-01-01-0.50` | 000000035 |
| 000000114 | 0.60 | `01-02-01-01-0.60` | **000000072** |
| 000000151 | 0.65 | `01-02-01-01-0.65` | 000000035 |
| 000000037 | **0.70** | `01-02-01-01-0.70` | 000000035 |
| 000000113 | 0.90 | `01-02-01-01-0.90` | 000000035 |

**Материал 0.60 (000000114) находится в другой группе** (000000072). Если логика `НахождениеМатериалаПоТипу` или `КорректировкаТолщиныМатериалаССайта` ищет материалы по **точному совпадению типа**, а не по родителю, то для артикула `01-02-01-01-0.70` родитель будет определён как `000000035`. Поиск «родственных» материалов по родителю `000000035` даст толщины: 0.50, 0.55, 0.65, 0.70, 0.80, 0.90, 1.00, 1.20. Значение 0.70 **присутствует** (000000037).

Почему же система не находит 0.70? **Возможные причины:**
1. При загрузке через XML материал `01-02-01-01-0.70` определяется некорректно (например, `НахождениеМатериалаПоТипу` не находит точное совпадение и возвращает пустую ссылку или другой материал).
2. Функция `КорректировкаТолщиныМатериалаССайта` использует не `Родитель` найденного материала, а **предопределённую группу** из настроек продукта. Если для продукта `1-1-1` настроена группа, в которой нет толщины 0.70, то функция падает на 0.50.
3. Функция `КорректировкаТолщиныМатериалаССайта` проверяет толщину по строке, а не по числу. В XML `0.7` — это строка. Если внутри функции есть сравнение `"0.7" = "0.70"`, оно вернёт `Ложь`.

## 6. Конкретные рекомендации по исправлению XML

### Рекомендация 1: Использовать тип-префикс в `<material>` (обязательно)

**Вместо:**
```xml
<material>01-02-01-01-0.70</material>
<thickness>0.7</thickness>
```

**Использовать:**
```xml
<material>01-02-01-01</material>
<thickness>0.7</thickness>
```

**Обоснование:** В шаблоне `order_template.xml` и в документации `README_ORDER_LOAD.md` явно указано, что ожидается **артикул материала (тип или полный)**. Полный артикул с толщиной (`01-02-01-01-0.70`) может конфликтовать с логикой корректировки толщины, которая ожидает, что толщина придёт из отдельного поля `<thickness>`, а не будет «зашита» в артикул.

### Рекомендация 2: Использовать корректные ID соединений (обязательно)

**Вместо:**
```xml
<connection_2>0</connection_2>
<connection_3>0</connection_3>
```

**Использовать:**
- Для круглых продуктов (`1-1-1`, `1-1-2`, `12-1-3`): `2`
- Для прямоугольных продуктов (`1-2-1`): `8`

```xml
<!-- Круглый продукт -->
<connection_0>2</connection_0>
<connection_1>2</connection_1>
<connection_2>2</connection_2>
<connection_3>2</connection_3>

<!-- Прямоугольный продукт -->
<connection_0>6</connection_0>
<connection_1>6</connection_1>
<connection_2>8</connection_2>
<connection_3>8</connection_3>
```

### Рекомендация 3: Использовать толщину, которая точно есть в базе

По данным `catalog_data.json` для типа `01-02-01-01` (Рулон оц.) доступны толщины:

| Толщина | Артикул | Код |
|---------|---------|-----|
| 0.50 | `01-02-01-01-0.50` | 000000163 |
| 0.55 | `01-02-01-01-0.55` | 000000149 |
| 0.60 | `01-02-01-01-0.60` | 000000114 |
| 0.65 | `01-02-01-01-0.65` | 000000151 |
| **0.70** | `01-02-01-01-0.70` | **000000037** |
| 0.80 | `01-02-01-01-0.80` | 000000040 |
| 0.90 | `01-02-01-01-0.90` | 000000113 |
| 1.00 | `01-02-01-01-1.00` | 000000108 |
| 1.20 | `01-02-01-01-1.20` | 000000150 |

**Важно:** 0.70 существует в базе. Если система не находит её, проблема скорее всего не в отсутствии данных, а в **формате передачи артикула или в логике сопоставления**.

### Рекомендация 4: Тестировать сначала с продуктом `1-1-2` (спирально-навивной)

В рабочем заказе `order_839.json` продукт `1-1-2` используется с материалом `01-02-01-01-0.70` и толщиной `0.7`. Это **проверенная комбинация**, которая работает в UI.

Перед добавлением других продуктов (особенно `1-1-1` и `12-1-3`) убедитесь, что XML с одним продуктом `1-1-2` загружается корректно. Если и здесь толщина заменяется на 0.5 — проблема точно в формате `<material>` или в `<thickness>`.

### Рекомендация 5: Для товаров без толщины указывать `<thickness>0</thickness>`

Пустой элемент `<thickness></thickness>` приведёт к ошибке (как в `Ошибка_20260618113438`):
```
{асОбщийМодуль.Модуль(7863)}: Преобразование значения к типу Число не может быть выполнено
```

**Всегда указывайте `0` для товаров, где толщина не применима.**
